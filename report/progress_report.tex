\section{Brief Project Description}
\vspace{-4pt}

Syntax highlighters, code search engines, snippet managers and Q\&A sites like Stack Overflow all
need to know what language a piece of code is written in---usually from a short fragment with no
filename, because the filename is exactly what is lost when code is pasted into a chat or scraped
from a page \citep{alreshedy2018scc}. \textbf{This project takes a raw snippet and predicts which of
10 popular languages it is (Python, Java, C++, JavaScript, Go, C, Ruby, PHP, Rust, C\#), using
nothing but the characters.} Rule-based detectors do badly on short fragments
\citep{vandam2016software} and GitHub's Linguist mostly just reads the extension \citep{linguist},
so a learned model fills a real gap.

Deep learning suits this input and output for three reasons. What separates one language from
another is many small overlapping hints---keywords, operator spellings, bracket and indentation
habits---and a character-level CNN learns those itself instead of me hand-writing rules for 10
languages \citep{zhang2015character}. Those hints are short character patterns like \texttt{:=},
\texttt{fn} and \texttt{\$var}, which is exactly what a small-kernel convolution spots
\citep{kim2014convolutional}. And reading characters avoids a circular problem: a word-based model
needs a tokenizer, but you cannot pick one until you know the language.

\subsection{Data Processing}
\vspace{-4pt}
\textbf{Source.} Rosetta Code \citep{rosettacode} (via the \texttt{acmeism/RosettaCodeData} mirror)
solves the same tasks in many languages, and each solution sits in a folder named after its language,
so the labels come free. Table~\ref{tab:clean} lists every cleaning step and what it removed, in
enough detail for a classmate to redo (\texttt{src/build\_dataset.py}, seed 360). Two steps matter
most: \emph{removing near-duplicates}, because Rosetta often stores several almost-identical versions
of one solution; and \emph{splitting by task rather than by file}, so all snippets from one task stay
in one split. I save each snippet twice---\texttt{raw}, and \texttt{nocomment} with comments removed
(10{,}271 clear the 64-character floor)---to see whether the model reads the code or the comments.

\textbf{One cleaned training sample}, copied exactly from \texttt{raw\_train.jsonl}
(\textsc{Python}, task \texttt{Call-a-foreign-language-function}, 121 characters; the two
\texttt{\#} comments are what \texttt{nocomment} removes):
\begin{lstlisting}[language=Python]
import ctypes
libc = ctypes.CDLL("/lib/libc.so.6")
libc.strcmp("abc", "def")     # -1
libc.strcmp("hello", "hello") #  0
\end{lstlisting}

\textbf{Plan for never-before-seen data.} Instead of reusing a public dataset, I collected the final
test set myself (\texttt{src/scrape\_heldout.py}): \textbf{740 snippets from 24 real GitHub
repositories}, 74 per language, cleaned the same way. None has any link to Rosetta Code, and it is
deliberately a different \emph{kind} of code---real library code, not puzzle answers. I checked every
snippet against every training snippet: \textbf{none overlap}, and the model never trains or tunes
on it.

\textbf{Challenges.} (i) Rosetta has little PHP (645 files vs.\ Python's 3{,}054), and since I keep
the classes balanced, that shortage drags \emph{every} class down to 1{,}036---the main limit on
dataset size. (ii) I split by task assuming a per-snippet split would let the model see the same
solution twice. I measured it (\texttt{src/leakage\_experiment.py}) and I was wrong: it changes the
baseline by only $+0.20$ and the CNN not at all---removing near-duplicates had already solved it. I
kept the by-task split as the safer choice. (iii) My comment remover works line by line, so it wrongly deletes text inside strings containing
\texttt{//} or \texttt{\#}, and in C deletes \texttt{\#include}/\texttt{\#define}, which are real
code. That hurts the no-comment score rather than helping, so the 92.49 below is if anything too low
and my conclusion is safe. Fixed for the final.

\vspace{-6pt}
\subsection{Baseline Model}
\vspace{-4pt}

The baseline uses no deep learning: it counts character sequences 1 to 3 long, weights them by
TF-IDF, and feeds them to \textbf{Multinomial Naive Bayes} on the same splits
\citep{khasnabish2014detecting,ugurel2002whats}. Nothing to tune beyond that 1--3 choice, 43{,}573
features, 4 seconds to train. It is a real bar: \textbf{90.14\%}, where guessing gets 10\%. Simple
character counting already does most of the job, so the CNN has to earn the rest.

\textbf{Qualitative (baseline).} Its GitHub mistakes are about English, not code. It calls PHP
\textsc{C\#} \textbf{34 times out of 74} and gets PHP right only \textbf{26}, though it never mixes
those two up on Rosetta. Java is called Python 24 times and C is called Python 20 times, for the same
reason: when a snippet is mostly an English comment, a word-counting model drifts toward whichever
language had the most English in training. \textbf{Challenge:} no tuning fixes this, because the
problem is what the model looks at, not its settings. That is the gap the CNN closes.

\vspace{-6pt}
\subsection{Primary Model}
\label{sec:primary}
\vspace{-4pt}

The main model (Figure~\ref{fig:pipeline}): \texttt{Embedding}$(102,32)$ turns each character into
32 numbers; three convolutions run \emph{side by side} over 3, 5 and 7 characters (128 filters each,
then ReLU); each keeps only its strongest response (global max-pool); the three join into 384
numbers; \texttt{Dropout}(0.5) drops half during training; \texttt{Linear}$(384,10)$ plus softmax
gives the answer. Three widths side by side, rather than one deep stack, let it see 3-, 5- and
7-character clues at once. It has \textbf{68,938 trainable parameters} across 5 weight layers, trained from scratch with
cross-entropy loss and Adam \citep{kingma2015adam} (learning rate $10^{-3}$, batch 64, at most 30
epochs, stopping once validation accuracy has not improved for 6), in \textbf{35 seconds} on an Apple
M-series GPU. Validation loss stops improving around epoch 10 and the curves separate after
12---the model is starting to memorise---so early stopping halted at 25 and kept \textbf{epoch 19}.

I also built the bidirectional GRU I promised in the proposal (\texttt{src/gru.py}; 42{,}186
parameters). \textbf{It did not work out}---the most useful thing I have learned so far: 90.27\% on
test, no better than Naive Bayes, for \textbf{647 seconds of training, 18 times the CNN}. In
hindsight that makes sense: what gives a language away in 256 characters are short local patterns,
so a model built to track long-range order pays for something this task never uses.

\textbf{What happens without comments.} Deleting every comment costs the CNN just \textbf{0.25
points} (92.74 to 92.49), which means it is reading the \emph{structure of the code}, not the
English in the comments. The baseline does the opposite: take its comments away and its GitHub score
actually \emph{goes up} 2.30 points (79.59 to 81.89).

\textbf{Qualitative (CNN).} The model scores 9.9 points lower on GitHub than on Rosetta
(Figure~\ref{fig:cm}), and reading the mistakes explains why. They fall into two groups.
\emph{(1) Java called JavaScript, 30 times: licence headers.} \textbf{22 of those 30} start with an
Apache or copyright header, and on average only \textbf{17\% of the snippet is code}---the rest is
English inside a \texttt{/*~*/} block. Rosetta files never have these headers, so the model never
learned that a real file often opens with legal boilerplate that looks identical in every language.
It is not really wrong; it just has almost no code to look at. \emph{(2) PHP called Rust, 16 times: PHP 8 attributes.} \textbf{13 of those 16} contain
\texttt{\#[}, and the training data explains it: \texttt{\#[} appears in \textbf{0\% of PHP snippets
(0 of 753)} but \textbf{9\% of Rust (63 of 703)}, so the model sensibly learned \texttt{\#[} means
Rust. But modern PHP 8 writes attributes exactly the same way (\texttt{\#[Group('...')]}), and they
appear in \textbf{19\% of my GitHub PHP (14 of 74)}. Rosetta's PHP predates PHP 8, so the model never
saw it---a textbook case of test data not matching training data. PHP's F1 drops 0.97 to 0.66.

\textbf{Challenges.} (i) The GitHub test set took the most effort: my first attempt got only 34
snippets per language because the C repositories I chose were too small. The GRU's 18$\times$ cost
also made it impractical to tune. (ii) Apple's GPU does not give identical results twice, so a fixed
seed does not pin the number down: eight reruns landed between \textbf{91.0 and 92.7} on test
(average 92.2). Table~3 is the run I committed, but the ordering holds---the CNN beat the baseline in
\textbf{all 8}, the closest by 0.87---so trust the ranking, not the exact figure. \textbf{Can I finish in time?}\label{sec:feas} Yes. The data is collected, the baseline is beaten on
both test sets, and the pipeline reruns in under 3 minutes. The error analysis already says what to
do next: strip licence headers, add post-2015 PHP, and add the SCC dataset
\citep{alreshedy2018scc}.
